# 运维团队工作流程

## 阶段总览

| 阶段 | 名称 | 核心角色 | 主要产出 |
|------|------|----------|----------|
| Phase 1 | 现状评估 | ops-manager, sys-engineer | 现状评估报告、资源清单、风险列表 |
| Phase 2 | 方案规划 | ops-manager, net-engineer, dba | 实施方案、变更计划、回滚策略 |
| Phase 3 | 实施部署 | sys-engineer, net-engineer, dba, automation-engineer | 部署完成、配置文件、自动化脚本 |
| Phase 4 | 监控建设 | monitor-engineer, security-ops | 监控面板、告警规则、安全基线 |
| Phase 5 | 运行优化 | ops-manager + 全员 | 优化报告、SLA 达标确认、知识沉淀 |

---

## Phase 1: 现状评估

**负责角色：** ops-manager（主导）、sys-engineer（执行）

### 目标
- 全面梳理当前基础设施状态，包括服务器、网络、数据库、监控覆盖情况
- 识别性能瓶颈、安全风险、单点故障
- 建立资源台账与基线数据

### 流程
1. ops-manager 发起评估任务，明确评估范围与时间节点
2. sys-engineer 收集系统资源信息（CPU/内存/磁盘/网络利用率）
3. 汇总现有告警记录、故障历史、容量趋势
4. 输出《现状评估报告》，包含资源清单、风险矩阵、改进建议

### 完成条件
- [ ] 资源台账完整，覆盖所有生产环境资产
- [ ] 风险项已识别并按优先级排序
- [ ] 评估报告经 ops-manager 审核通过

---

## Phase 2: 方案规划

**负责角色：** ops-manager（主导）、net-engineer、dba

### 目标
- 根据评估结果制定可执行的运维改进方案
- 明确变更窗口、回滚策略、影响面分析
- 确定各角色的任务分工与时间线

### 流程
1. ops-manager 组织方案评审会，拉通各专业线需求
2. net-engineer 规划网络架构调整方案（如有需要）
3. dba 规划数据库优化/迁移方案（如有需要）
4. 编写《实施方案》与《变更计划》，明确每步操作与检查点
5. 制定回滚方案，确保每个变更可逆

### 完成条件
- [ ] 实施方案经全员评审并签字确认
- [ ] 变更窗口已确定，回滚策略已验证
- [ ] 任务分工表已分配到具体角色

---

## Phase 3: 实施部署

**负责角色：** sys-engineer、net-engineer、dba、automation-engineer

### 目标
- 按照方案执行基础设施变更与部署
- 通过自动化手段减少人为操作风险
- 确保每步变更有验证、有记录

### 流程
1. automation-engineer 准备 IaC 代码与自动化脚本
2. sys-engineer 执行系统层面变更（OS 配置、服务部署）
3. net-engineer 执行网络层面变更（防火墙、负载均衡、DNS）
4. dba 执行数据库层面变更（配置优化、迁移、副本搭建）
5. 每步完成后执行验证检查，记录变更日志

### 完成条件
- [ ] 所有变更已按方案执行完毕
- [ ] 每步变更有验证记录，无遗留问题
- [ ] 自动化脚本已入库，可重复执行

---

## Phase 4: 监控建设

**负责角色：** monitor-engineer（主导）、security-ops

### 目标
- 建立全链路监控覆盖，确保问题可被及时发现
- 配置合理的告警规则，避免告警疲劳
- 完成安全基线检查与加固

### 流程
1. monitor-engineer 部署/更新监控采集器与面板
2. 配置分级告警规则（P0-P3）与通知渠道
3. security-ops 执行安全基线扫描与加固
4. 建立日志聚合与审计追踪机制
5. 编写《监控与告警手册》

### 完成条件
- [ ] 监控覆盖率达到目标（核心服务 100%）
- [ ] 告警规则已验证，无误报/漏报
- [ ] 安全基线检查通过，加固项已闭环

---

## Phase 5: 运行优化

**负责角色：** ops-manager（主导）+ 全员

### 目标
- 持续观察运行状态，优化性能与成本
- 沉淀运维知识，完善标准化流程
- 确认 SLA 指标达标

### 流程
1. ops-manager 组织周/月运维复盘会议
2. 各角色提交所辖领域的优化建议
3. 执行性能调优、成本优化、流程改进
4. 更新运维文档与知识库
5. 输出《运行优化报告》与 SLA 达标确认

### 完成条件
- [ ] SLA 指标达标且持续稳定
- [ ] 运维知识库已更新，关键操作有 Runbook
- [ ] 优化报告经 ops-manager 审核归档

---

## 跨阶段规则

1. **变更控制：** 所有生产环境变更必须经过审批，有变更单记录
2. **回滚优先：** 每个变更必须有回滚方案，执行前确认回滚可行性
3. **灰度发布：** 批量变更采用分批执行策略，观察无异常后再扩大范围
4. **文档同步：** 每阶段产出物必须及时归档，配置变更同步更新 CMDB
5. **安全前置：** 所有方案需经 security-ops 审核安全合规性
6. **自动化优先：** 可自动化的操作禁止手工执行，减少人为错误
7. **阶段门禁：** 上一阶段完成条件全部满足后，方可进入下一阶段
8. **应急通道：** 紧急故障可跳过常规流程，但事后必须补齐文档与复盘
