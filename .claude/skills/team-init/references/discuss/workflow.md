# 讨论/研讨团队工作流程

> 讨论型团队与执行型团队截然不同：非线性、轮次驱动、以共识为目标。
> 所有阶段围绕"达成高质量结论"展开，而非线性交付。

---

## 阶段总览

| 阶段 | 名称 | 核心目标 | 激活角色 |
|------|------|----------|----------|
| Phase 0 | 议题设定 | 分解议题、确定顺序 | moderator |
| Phase 1 | 观点征集 | 收集各方专业观点 | moderator, domain-expert, critic |
| Phase 2 | 交叉讨论 | 相互回应、深化分析 | 全体 |
| Phase 3 | 汇总评估 | 提炼共识与分歧 | moderator, synthesizer |
| Phase 4 | 结论确认 | 确认结论或裁定 | moderator, synthesizer |
| Phase 5 | 产出整合 | 生成汇总文档 | moderator, synthesizer, recorder |

---

## Phase 0: 议题设定

<phase id="0" name="议题设定">

**目标：** moderator 分析项目描述，将讨论范围拆解为可独立讨论的议题，确定讨论顺序，为后续轮次循环做好准备。

**激活角色：** moderator

**流程：**
1. moderator 阅读项目描述，识别核心讨论范围与目标
2. 将讨论范围拆解为若干独立议题，每个议题包含：
   - 议题标题（简短、明确）
   - 背景描述（为什么要讨论这个议题）
   - 期望产出类型：技术决策 / 需求研讨 / 头脑风暴
3. 使用 TaskCreate 为每个议题创建独立任务，状态设为 pending
4. 确定议题讨论顺序（优先级高或存在依赖关系的议题先讨论）
5. 将首个议题的任务状态更新为 in_progress
6. 广播议题列表给所有成员，确认各方已知悉

**完成条件：**
- [ ] 议题列表已确认，每个议题有明确标题、背景和产出类型
- [ ] 每个议题均已通过 TaskCreate 创建对应任务
- [ ] 首个议题任务已标记为 in_progress

</phase>

---

## Phase 1: 观点征集（每轮重复）

<phase id="1" name="观点征集">

**目标：** 各成员围绕当前议题（或上轮残余分歧点）独立提交观点，确保所有立场和论据完整呈现。

**激活角色：** moderator, domain-expert（一个或多个）, critic（可选）

**流程：**
1. moderator 广播议题启动消息：
   - 首轮：发送 `[TOPIC]` 消息，包含议题标题、背景描述、本轮讨论焦点
   - 后续轮次：发送 `[TOPIC]` 消息，仅列出上轮未消除的分歧点，缩小讨论范围
2. 各 domain-expert 独立思考后，各自提交 `[OPINION]` 消息，包含：
   - 立场（支持/反对/中立/提案）
   - 核心论据（事实依据或推理过程）
   - 潜在风险或局限性
3. critic（如有）额外提交质疑性 `[OPINION]`，专注于识别论据漏洞和隐藏风险
4. 每位成员仅提交一次观点，本阶段不允许相互回应

**完成条件：**
- [ ] 所有激活的 domain-expert 均已提交 `[OPINION]`
- [ ] critic（如有）已提交质疑性观点

</phase>

---

## Phase 2: 交叉讨论（每轮重复）

<phase id="2" name="交叉讨论">

**目标：** 成员针对他人观点进行回应，通过交叉验证深化分析，暴露真正的分歧点。

**激活角色：** 全体成员

**流程：**
1. moderator 将本轮所有原始观点汇编后，以 `[REVIEW]` 消息广播给全体成员
2. 各成员阅读全部原始观点后，提交 `[RESPONSE]` 消息：
   - 明确指出回应对象（@某成员的观点）
   - 表明态度：赞同 / 反对 / 补充延伸
   - 赞同：说明赞同理由，可补充支持论据
   - 反对：提出具体的反驳论据，不接受无理由反对
   - 补充：在他人观点基础上延伸，扩展分析角度
3. 每位成员本轮限提交 1-2 条 `[RESPONSE]`，避免无限循环
4. recorder（如有）实时将讨论内容追加记录至讨论文档

**完成条件：**
- [ ] 所有激活成员均已完成回应（或明确声明无补充）
- [ ] 每条 `[RESPONSE]` 均有明确回应对象和态度

</phase>

---

## Phase 3: 汇总评估（每轮重复）

<phase id="3" name="汇总评估">

**目标：** 提炼本轮讨论的共识与分歧，判断是否已收敛，决定进入结论阶段还是继续下一轮。

**激活角色：** moderator, synthesizer（可选）

**流程：**
1. synthesizer（如有）提交本轮 `[SUMMARY]` 消息，包含：
   - 已达成共识的观点列表
   - 仍存在分歧的焦点列表（具体说明分歧内容）
   - 本轮新发现或新信息
   - 收敛评估：分歧是否在减少
2. 若无 synthesizer，由 moderator 自行完成上述汇总
3. moderator 依据汇总结果作出收敛判断：
   - **分歧点 = 0** → 宣布收敛，进入 Phase 4
   - **分歧点 > 0 且当前轮次 < max_rounds** → 回到 Phase 1，仅针对剩余分歧点展开新一轮讨论
   - **当前轮次 = max_rounds** → 宣布达到最大轮次，进入裁定流程后转入 Phase 4

**完成条件：**
- [ ] `[SUMMARY]` 已提交，共识与分歧明确列出
- [ ] 收敛性判断已完成，后续路径已确定

</phase>

---

## Phase 4: 结论确认（每个议题）

<phase id="4" name="结论确认">

**目标：** 对当前议题形成最终结论，生成产出文档，标记议题完成，推进至下一议题。

**激活角色：** moderator, synthesizer

**流程：**
1. **共识达成路径：**
   - moderator 宣布共识结论，简述支撑论据
   - synthesizer 将结论整理为正式产出文档（格式见"产出物格式概览"）
2. **裁定路径（达到 max_rounds）：**
   - moderator 发送 `[DECISION]` 消息，包含：最终结论、采纳理由、被否定立场的处理方式、风险接受声明
   - moderator 通知全体成员裁定结果
   - synthesizer 将裁定结论整理为正式产出文档
3. 将当前议题对应的任务状态更新为 completed
4. 检查是否还有 pending 状态的议题：
   - **有剩余议题** → 将下一个议题任务更新为 in_progress，回到 Phase 1 开始新议题讨论
   - **无剩余议题** → 进入 Phase 5

**完成条件：**
- [ ] 当前议题的产出文档已生成
- [ ] 议题任务已标记为 completed
- [ ] 后续路径已确定（下一议题或进入 Phase 5）

</phase>

---

## Phase 5: 产出整合

<phase id="5" name="产出整合">

**目标：** 将所有议题的结论整合为完整的汇总文档，完成讨论记录归档。

**激活角色：** moderator, synthesizer, recorder

**流程：**
1. synthesizer 生成汇总文档 `docs/discussions/{project_name}-summary.md`，包含：
   - 讨论背景与参与成员
   - 各议题标题及最终结论（一句话概括）
   - 结论达成方式：共识 / 裁定（注明轮次）
   - 待跟进事项（讨论中识别的后续行动项）
   - 重大分歧记录（被裁定否决的立场及其原始论据）
2. recorder（如有）完成完整讨论过程的记录归档，确保每轮的 `[OPINION]`、`[RESPONSE]`、`[SUMMARY]` 均已留存
3. moderator 审阅汇总文档，确认内容完整准确后宣布讨论结束

**完成条件：**
- [ ] `docs/discussions/{project_name}-summary.md` 已生成且内容完整
- [ ] 完整讨论记录已归档
- [ ] moderator 确认讨论正式结束

</phase>

## 通信协议

所有成员通信须使用以下标准消息前缀，以便 moderator 和 recorder 识别消息类型：

| 消息前缀 | 发送者 | 接收者 | 格式要求 |
|----------|--------|--------|----------|
| `[TOPIC]` | moderator | 全体 | 议题标题 + 本轮讨论焦点（首轮含背景，后续轮次仅列分歧点） |
| `[OPINION]` | domain-expert, critic | moderator | 立场（一词）+ 核心论据 + 风险说明 |
| `[REVIEW]` | moderator | 全体 | 本轮所有原始观点的汇编列表，不加评论 |
| `[RESPONSE]` | 全体成员 | moderator | @回应对象 + 态度（赞同/反对/补充）+ 具体说明 |
| `[SUMMARY]` | synthesizer / moderator | 全体 | 共识列表 + 分歧列表 + 新发现 + 收敛评估 |
| `[DECISION]` | moderator | 全体 | 最终结论 + 采纳理由 + 风险接受声明 |

---

## 收敛判断规则

moderator 在 Phase 3 结束时依据以下标准判断收敛性：

- **分歧点消除标准：** 原先持不同立场的成员，在当前轮次的 `[RESPONSE]` 中明确表示"接受"或"理解并不再反对"对方观点，则该分歧点视为消除
- **未消除分歧：** 仍有成员坚持不同立场，且未能提出对方认可的新论据，该分歧点继续列入下一轮
- **触发裁定条件：** 当轮次计数达到 max_rounds（由 moderator 在 Phase 0 设定，默认为 3 轮），无论分歧是否消除，均进入裁定流程

---

## 裁定流程

当轮次达到 max_rounds 时，moderator 执行以下步骤：

1. 列出所有未消除的分歧点及各方立场
2. 逐一权衡各立场的论据质量、风险程度和影响范围
3. 做出最终决策，选择其中一个立场或提出综合方案
4. 发送 `[DECISION]` 消息，明确标注：决策内容、采纳理由、被否定立场的处理（接受/暂缓/记录保留）
5. 通知全体成员，给予 1 次简短异议表达机会（不影响决策执行）

---

## 产出物格式概览

根据议题的期望产出类型，synthesizer 选择对应格式（详细模板见 `synthesizer.md`）：

| 产出类型 | 内容要点 | 典型结构 |
|----------|----------|----------|
| **技术决策** | 决策结论、备选方案对比、选型理由、风险承接 | 决策记录（ADR 格式） |
| **需求研讨** | 需求范围、约束条件、优先级、待明确事项 | 需求说明文档 |
| **头脑风暴** | 所有提案列表、初步评分、推荐优先探索的方向 | 创意清单 + 评估矩阵 |

---

## 跨议题规则

- **信息传递：** 前一个议题的结论自动成为后续议题的已知输入，后续议题讨论时无需重新论证已确认的结论
- **依赖处理：** 若议题 B 依赖议题 A 的结论，则议题 A 必须先完成；Phase 0 设定顺序时应识别此类依赖
- **结论修订：** 若后续议题讨论中发现前一议题的结论存在问题，moderator 可宣布回退，将对应议题任务重新设为 in_progress 并重新讨论
- **上下文共享：** recorder 维护的讨论记录文档对所有成员可读，成员可在任意时刻引用历史轮次的论据
