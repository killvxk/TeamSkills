# 软件测试团队工作流程

## 阶段总览

| 阶段 | 名称 | 核心角色 | 主要产出 |
|------|------|----------|----------|
| Phase 1 | 测试策略制定 | test-manager + test-architect | 测试策略文档、测试计划 |
| Phase 2 | 测试用例设计 | test-architect + functional-tester | 测试用例集、测试数据 |
| Phase 3 | 测试执行 | functional-tester + perf-tester + security-tester | 测试报告、缺陷列表 |
| Phase 4 | 自动化建设 | automation-tester | 自动化框架、CI/CD 集成 |
| Phase 5 | 测试报告与收尾 | test-manager | 最终测试报告、质量评估 |

---

## Phase 1: 测试策略制定

<phase id="1" name="测试策略制定">

### 目标

- 明确被测系统的范围、质量目标和风险点
- 制定测试策略，确定测试类型、测试层级和优先级
- 规划测试资源、工具和环境需求
- 建立质量度量标准和准入/准出条件

### 核心角色

- **test-manager**: 主导测试计划制定，协调资源分配
- **test-architect**: 设计测试架构，选择测试工具和框架

### 流程

1. test-manager 分析项目需求文档和设计文档，识别测试范围
2. test-architect 评估系统架构，确定技术风险和测试难点
3. test-manager 与 test-architect 协作制定测试策略文档
4. 确定测试类型分配：功能测试、性能测试、安全测试的比例和优先级
5. 规划测试环境和测试数据需求
6. 制定测试进度计划和里程碑

### 产出物

- 测试策略文档（测试范围、测试类型、风险评估）
- 测试计划（进度、资源、环境需求）
- 质量度量标准（准入/准出条件、覆盖率目标）

### 完成条件

- [ ] 测试策略文档已评审通过
- [ ] 测试计划已获得相关方确认
- [ ] 测试环境需求已明确
- [ ] 各测试角色的任务分配已完成

</phase>

---

## Phase 2: 测试用例设计

<phase id="2" name="测试用例设计">

### 目标

- 基于需求和设计文档，设计覆盖全面的测试用例
- 建立测试用例管理体系和评审机制
- 准备测试数据和前置条件
- 确保测试用例的可执行性和可维护性

### 核心角色

- **test-architect**: 设计测试框架结构，指导用例编写规范
- **functional-tester**: 编写功能测试用例，设计测试场景

### 流程

1. test-architect 制定测试用例编写规范和模板
2. functional-tester 根据需求文档编写功能测试用例
3. test-architect 审查用例覆盖度，补充边界条件和异常场景
4. 设计测试数据集，覆盖正常流程和异常流程
5. 用例交叉评审，确保可执行性
6. perf-tester 和 security-tester 分别准备性能测试和安全测试的场景设计

### 产出物

- 功能测试用例集（含优先级标记）
- 测试数据集
- 测试场景矩阵（需求追溯矩阵）
- 性能测试场景设计初稿
- 安全测试检查清单初稿

### 完成条件

- [ ] 功能测试用例覆盖率达到策略要求
- [ ] 测试用例已通过评审
- [ ] 测试数据已准备就绪
- [ ] 需求追溯矩阵完整，无遗漏需求

</phase>

---

## Phase 3: 测试执行

<phase id="3" name="测试执行">

### 目标

- 按计划执行各类测试，发现并记录缺陷
- 进行功能验证、性能评估和安全检查
- 跟踪缺陷修复和回归验证
- 收集测试度量数据

### 核心角色

- **functional-tester**: 执行功能测试、回归测试、探索性测试
- **perf-tester**: 执行性能测试、负载测试、压力测试
- **security-tester**: 执行安全扫描、渗透测试、合规检查

### 流程

1. functional-tester 按优先级执行功能测试用例
2. 发现缺陷后提交缺陷报告，标记严重级别和优先级
3. perf-tester 并行执行性能基准测试和负载测试
4. security-tester 并行执行漏洞扫描和安全合规检查
5. 缺陷修复后执行回归测试，验证修复效果
6. functional-tester 进行探索性测试，发现用例未覆盖的问题
7. 各角色定期向 test-manager 汇报测试进度和阻塞项

### 产出物

- 测试执行记录（通过/失败/阻塞统计）
- 缺陷报告列表（含严重级别、复现步骤）
- 性能测试报告（响应时间、吞吐量、资源使用率）
- 安全测试报告（漏洞列表、风险等级、修复建议）
- 回归测试结果

### 完成条件

- [ ] 所有高优先级测试用例已执行
- [ ] 严重和高优先级缺陷已修复并回归通过
- [ ] 性能指标满足预定目标
- [ ] 无高危安全漏洞遗留
- [ ] 测试通过率达到准出条件

</phase>

## Phase 4: 自动化建设

<phase id="4" name="自动化建设">

### 目标

- 将高价值的手工测试用例转化为自动化脚本
- 搭建自动化测试框架，集成到 CI/CD 流水线
- 提升回归测试效率，减少重复劳动
- 建立自动化测试的维护和更新机制

### 核心角色

- **automation-tester**: 主导自动化框架搭建和脚本编写

### 流程

1. automation-tester 根据测试用例优先级和稳定性，选择自动化候选用例
2. 搭建或完善自动化测试框架（UI 自动化、API 自动化、单元测试辅助）
3. 编写自动化测试脚本，遵循 Page Object 等设计模式
4. 将自动化测试集成到 CI/CD 流水线
5. 配置定时执行和触发规则
6. 编写自动化测试维护文档

### 产出物

- 自动化测试框架代码
- 自动化测试脚本集
- CI/CD 配置文件
- 自动化覆盖率报告
- 自动化维护指南

### 完成条件

- [ ] 核心回归用例已实现自动化
- [ ] 自动化测试可在 CI/CD 中稳定运行
- [ ] 自动化测试通过率 > 95%
- [ ] 维护文档已编写完成

</phase>

---

## Phase 5: 测试报告与收尾

<phase id="5" name="测试报告与收尾">

### 目标

- 汇总所有测试结果，编写最终测试报告
- 评估产品质量，给出发布建议
- 总结测试过程中的经验教训
- 归档测试资产，为后续迭代提供参考

### 核心角色

- **test-manager**: 主导报告编写、质量评估和项目收尾

### 流程

1. test-manager 汇总各类测试报告和度量数据
2. 分析缺陷分布、测试覆盖率、质量趋势
3. 评估剩余风险和已知问题的影响
4. 编写最终测试报告，包含发布建议
5. 组织测试回顾会议，总结经验教训
6. 归档测试用例、测试数据、测试报告等资产

### 产出物

- 最终测试报告（质量评估、风险分析、发布建议）
- 测试度量报告（覆盖率、缺陷密度、测试效率）
- 经验教训总结
- 测试资产归档

### 完成条件

- [ ] 最终测试报告已完成并通过评审
- [ ] 发布建议已提交给项目相关方
- [ ] 经验教训已记录
- [ ] 测试资产已归档

</phase>

## 跨阶段规则

### 沟通与协作

- 所有测试角色通过 `SendMessage` 工具进行跨角色沟通
- test-manager 负责协调各角色间的依赖和阻塞问题
- 缺陷报告必须包含：复现步骤、预期结果、实际结果、严重级别
- 阻塞性问题需立即上报 test-manager

### 质量门禁

- 每个阶段必须满足完成条件后才能进入下一阶段
- 测试用例变更需经过评审
- 缺陷修复后必须进行回归验证
- 自动化测试失败必须在 24 小时内修复或标记

### 文档与追溯

- 所有测试产出物必须版本化管理
- 测试用例必须关联到对应需求（需求追溯）
- 缺陷必须关联到对应测试用例
- 测试报告必须包含数据支撑，避免主观判断

### 工具与环境

- 测试环境独立于开发环境，由 test-manager 协调管理
- 测试数据不得使用生产真实数据（脱敏处理）
- 自动化测试必须支持并行执行和环境隔离
- 安全测试必须在授权范围内进行，禁止未授权的渗透操作
