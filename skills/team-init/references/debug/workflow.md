# 调试/Bug修复 团队工作流程

## 阶段总览

| 阶段 | 名称 | 负责角色 | 关键产出 |
|------|------|----------|----------|
| Phase 1 | 问题收集与分类 | debug-lead, issue-analyst | 问题优先级列表、分类报告 |
| Phase 2 | 问题复现 | issue-analyst | 复现步骤、环境报告、日志分析 |
| Phase 3 | 根因分析 | root-cause-analyst | 根因报告、分析链路 |
| Phase 4 | 修复实现 | fix-engineer, code-reviewer | 修复补丁、单元测试、审查意见 |
| Phase 5 | 回归验证 | regression-tester, debug-lead | 回归测试套件、验证报告 |

> **注意：** 阶段 2-4 之间存在频繁的迭代循环。当根因分析发现新线索时可能需要回到阶段 2 重新复现；当修复实现发现根因不准确时需要回到阶段 3 重新分析。

---

## Phase 1: 问题收集与分类（Issue Triage）

<phase id="1">

**目标：** 收集所有待处理问题，按严重程度和影响范围进行分类排序，确定处理优先级。

**流程：**
1. debug-lead 汇总所有上报的问题（用户反馈、监控告警、日志异常等）
2. issue-analyst 对每个问题进行初步分析，收集基本信息（错误信息、发生频率、影响用户数）
3. debug-lead 与 issue-analyst 共同评估优先级（P0-紧急/P1-高/P2-中/P3-低）
4. debug-lead 将问题分配给对应角色，启动后续阶段

**完成条件：**
- 所有问题已分类并标注优先级
- 每个问题有明确的负责人和预期时间线
- P0/P1 问题已立即进入 Phase 2

</phase>

## Phase 2: 问题复现（Reproduction）

<phase id="2">

**目标：** 稳定复现问题，确定触发条件和环境依赖，为根因分析提供可靠基础。

**流程：**
1. issue-analyst 根据问题描述搭建复现环境
2. 按照报告的步骤尝试复现，记录每一步的系统状态
3. 分析相关日志、监控数据、错误堆栈
4. 编写最小复现步骤（Minimal Reproduction Steps）
5. 确认复现的稳定性（必现 / 概率性 / 特定条件触发）

**完成条件：**
- 问题可稳定复现，或已明确概率性复现的条件
- 最小复现步骤已文档化
- 环境信息已完整记录（操作系统、版本、配置等）

</phase>

## Phase 3: 根因分析（Root Cause Analysis）

<phase id="3">

**目标：** 定位问题的根本原因，而非表面症状，输出明确的根因报告。

**流程：**
1. root-cause-analyst 基于复现步骤重现问题
2. 使用科学调试法：提出假设 → 设计验证实验 → 执行验证 → 确认或排除
3. 追踪代码执行路径，定位异常发生的精确位置
4. 必要时使用二分法（git bisect）定位引入问题的提交
5. 编写根因报告，说明问题机制和影响范围

**完成条件：**
- 根因已明确定位到具体代码位置和逻辑缺陷
- 根因报告已通过 debug-lead 审核
- 已评估问题的影响范围（是否有其他类似隐患）

</phase>

## Phase 4: 修复实现（Fix Implementation）

<phase id="4">

**目标：** 实现最小化的精准修复，确保不引入新问题。

**流程：**
1. fix-engineer 基于根因报告设计修复方案
2. 实现最小修复补丁（遵循最小变更原则）
3. 编写针对该问题的单元测试（确保问题不会再次出现）
4. code-reviewer 审查修复代码（正确性、副作用、编码规范、测试覆盖）
5. 根据审查意见修改，直到审查通过

**完成条件：**
- 修复补丁已通过代码审查
- 针对该问题的单元测试已编写并通过
- 副作用分析已完成，确认无新增风险

</phase>

## Phase 5: 回归验证（Regression Testing）

<phase id="5">

**目标：** 验证修复有效且未破坏现有功能。

**流程：**
1. regression-tester 设计回归测试用例（覆盖修复点及关联功能）
2. 执行完整回归测试套件
3. 验证原始问题已修复（使用 Phase 2 的复现步骤验证）
4. 验证相关功能未受影响
5. debug-lead 审核测试结果，做出发布决策

**完成条件：**
- 原始问题已验证修复
- 回归测试全部通过
- debug-lead 确认可以发布

</phase>

## 跨阶段规则

<cross_phase_rules>

1. **迭代回退规则：** Phase 2-4 之间允许自由回退。回退时必须记录原因和新发现。
2. **信息传递规则：** 每个阶段的产出物必须完整传递给下一阶段，不得遗漏关键信息。
3. **阻塞升级规则：** 任何阶段阻塞超过预期时间的 50%，必须上报 debug-lead 重新评估。
4. **并行处理规则：** 多个独立问题可并行处理，但同一问题的各阶段必须串行推进。
5. **文档同步规则：** 每个阶段完成后，相关文档必须在进入下一阶段前更新完毕。
6. **紧急问题快速通道：** P0 问题可跳过部分文档要求，但修复后必须补齐。
7. **知识沉淀规则：** 每个修复完成后，将根因和解决方案记录到知识库，避免同类问题重复分析。

</cross_phase_rules>
