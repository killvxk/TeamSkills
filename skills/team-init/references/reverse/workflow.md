# 逆向工程团队工作流程

> **重要说明：** 与开发工作流不同，逆向工程的各阶段可以重叠和迭代。发现新信息后可能需要回退到之前的阶段重新分析。

## 阶段总览

| 阶段 | 名称 | 负责角色 | 主要产出 |
|------|------|----------|----------|
| Phase 1 | 侦查与信息收集 | re-lead + 全员 | 目标画像、攻击面清单、初步分析计划 |
| Phase 2 | 静态分析 | static-analyst | 反汇编注释、函数映射、数据结构定义 |
| Phase 3 | 动态分析 | dynamic-analyst | 执行轨迹、内存快照、行为报告 |
| Phase 4 | 深度分析与漏洞研究 | vuln-researcher + protocol-analyst | 漏洞报告、协议规范、PoC 代码 |
| Phase 5 | 文档编写与报告 | documenter + re-lead | 最终分析报告、技术文档、架构图 |

---

## Phase 1: 侦查与信息收集

<phase id="1" name="侦查与信息收集">

### 目标
- 确定分析目标的基本属性（架构、平台、编译器、保护机制）
- 收集公开信息（文档、已知漏洞、相关研究）
- 制定分析策略与任务分配计划

### 流程
1. **目标识别** — 使用 `file`、`binwalk`、`readelf`/`PE Explorer` 等工具确认文件类型与架构
2. **信息收集** — 搜索目标相关的公开文档、CVE、技术文章
3. **保护机制检测** — 检查 ASLR、DEP/NX、栈保护、代码签名、加壳/混淆
4. **攻击面评估** — 识别输入点、网络接口、IPC 通道、文件解析逻辑
5. **计划制定** — re-lead 根据收集的信息制定详细分析计划并分配任务

### 完成条件
- [ ] 目标画像文档完成（架构、平台、保护机制、依赖关系）
- [ ] 攻击面清单已列出并按优先级排序
- [ ] 分析计划已制定，任务已分配到各角色
- [ ] 所有团队成员已了解目标背景与分析策略

</phase>

## Phase 2: 静态分析

<phase id="2" name="静态分析">

### 目标
- 理解程序整体结构与控制流
- 识别关键函数、数据结构与算法
- 为动态分析提供切入点与断点建议

### 流程
1. **反汇编与反编译** — 使用 IDA Pro / Ghidra 加载目标，进行自动分析
2. **字符串分析** — 提取并分类字符串常量，识别关键路径
3. **导入导出分析** — 分析 API 调用模式，识别系统交互方式
4. **控制流分析** — 构建函数调用图，识别关键路径与分支逻辑
5. **数据结构恢复** — 识别并定义关键数据结构与类层次
6. **注释与标注** — 对关键函数与数据结构添加注释，重命名符号

### 完成条件
- [ ] 主要函数已识别并注释
- [ ] 关键数据结构已定义
- [ ] 函数调用关系图已生成
- [ ] 可疑代码区域已标记并通知相关角色
- [ ] 动态分析断点建议已提交给 dynamic-analyst

</phase>

## Phase 3: 动态分析

<phase id="3" name="动态分析">

### 目标
- 验证静态分析的推断
- 观察程序运行时行为与数据流
- 捕获运行时数据（内存布局、寄存器状态、系统调用）

### 流程
1. **环境搭建** — 配置安全的分析环境（虚拟机/沙箱）
2. **断点调试** — 根据静态分析结果设置断点，跟踪执行流
3. **系统调用追踪** — 使用 strace/ltrace/Process Monitor 记录系统交互
4. **内存分析** — 捕获关键时刻的内存快照，分析数据布局
5. **Hook 与插桩** — 使用 Frida 等工具注入分析代码，监控特定函数
6. **结果反馈** — 将发现反馈给 static-analyst 修正分析结果

### 完成条件
- [ ] 关键执行路径已跟踪验证
- [ ] 运行时数据结构布局已确认
- [ ] 系统交互行为已记录
- [ ] 静态分析中的疑点已验证或修正
- [ ] 异常行为已记录并通知 vuln-researcher

</phase>

## Phase 4: 深度分析与漏洞研究

<phase id="4" name="深度分析与漏洞研究">

### 目标
- 深入分析通信协议与自定义格式
- 识别安全漏洞并评估影响
- 开发验证性 PoC（在授权范围内）

### 流程
1. **协议逆向** — protocol-analyst 抓包分析，重建消息格式与状态机
2. **漏洞识别** — vuln-researcher 根据前序分析结果寻找安全缺陷
3. **根因分析** — 确定漏洞的技术根因与触发条件
4. **影响评估** — 评估漏洞的严重程度与影响范围
5. **PoC 开发** — 在授权范围内编写最小化验证代码
6. **交叉验证** — 团队内部复核发现的漏洞与协议规范

### 完成条件
- [ ] 协议规范文档已完成（如适用）
- [ ] 已识别的漏洞均有详细报告
- [ ] PoC 代码已编写并验证（如适用）
- [ ] 影响评估已完成
- [ ] 所有发现已经 re-lead 审核确认

</phase>

## Phase 5: 文档编写与报告

<phase id="5" name="文档编写与报告">

### 目标
- 整合所有分析结果为结构化文档
- 生成最终分析报告
- 建立知识库以供后续参考

### 流程
1. **素材收集** — documenter 从各角色收集分析结果与原始数据
2. **报告编写** — 按标准模板编写分析报告
3. **图表绘制** — 绘制架构图、流程图、协议状态图
4. **审核校对** — re-lead 审核报告的技术准确性与完整性
5. **知识归档** — 将分析成果归档到知识库

### 完成条件
- [ ] 最终分析报告已完成并通过审核
- [ ] 所有技术文档已归档
- [ ] 架构图与流程图已绘制
- [ ] 知识库已更新
- [ ] 团队复盘总结已记录

</phase>

## 跨阶段规则

### 迭代与回退
- 任何阶段的新发现都可能触发回退到之前阶段重新分析
- 动态分析结果必须反馈给静态分析进行修正
- 漏洞研究中的新发现可能需要额外的静态/动态分析支持

### 信息共享
- 所有分析产出物必须及时共享到团队共享空间
- 关键发现必须立即通知 re-lead 和相关角色
- 使用统一的命名规范和注释格式

### 安全与合规
- 所有分析活动必须在合法授权范围内进行
- 敏感发现按团队安全策略处理
- 分析环境必须隔离，防止意外泄漏或损害

### 质量控制
- 每个阶段的产出物必须经过至少一位其他角色的交叉审核
- re-lead 对所有关键发现拥有最终审核权
- 不确定的结论必须明确标注置信度
